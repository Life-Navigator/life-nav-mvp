# LIOS — Risk Lifecycle

> The complete lifecycle of a **risk** (and its sibling, an opportunity) in LifeNavigator: detection,
> evidence, surfacing, tracking, resolution. Validation review against the LIOS architecture. Architecture
> review only — no code, no prompts.

A "risk" = an evidence-backed statement that something in the user's life could go wrong if unaddressed
(e.g. "underinsured for income replacement," "no emergency fund"). Opportunities are the positive mirror and
follow the same lifecycle. Both are **recommendations** of `rec_type` RISK / OPPORTUNITY and therefore
inherit the `RECOMMENDATION_LIFECYCLE.md` write contract — this document adds the risk-specific rules.

---

## 1. The cardinal rule (learned the hard way, enforced)

> **A risk may only exist if it is generated by the recommendation engine from real evidence.** Risks are
> never templated, never archetype-derived, never invented by the LLM, and never a generic label.

This is live: `discover_goal` no longer auto-creates archetype risks/opportunities; the Life Model grounds
risks/opps from the recommendation engine only; generic-label gates (`GENERIC_RISK_OPP_LABELS`) prevent
template phrases from leaking onto the dashboard. The validator forbids the advisor from inventing risks.

---

## 2. States

| State           | Meaning                                                      | Visible?           |
| --------------- | ------------------------------------------------------------ | ------------------ |
| `detected`      | the engine found a candidate risk from evidence              | no (internal)      |
| `evidenced`     | has ≥1 evidence statement with a `source_table`              | eligible           |
| `ranked`        | scored for severity × likelihood × impact                    | eligible           |
| `surfaced`      | shown to the user (dashboard / Life Model / advisor context) | yes                |
| `acknowledged`  | user has seen/accepted it                                    | yes                |
| `mitigating`    | user took an action linked to it                             | yes                |
| `resolved`      | the underlying condition no longer holds (evidence changed)  | yes (history)      |
| `dismissed`     | user explicitly dismissed it                                 | hidden (tombstone) |
| `expired/stale` | evidence aged past freshness                                 | re-evaluate        |

---

## 3. State transitions

```
 evidence ─▶ detected ─(evidence present?)─▶ evidenced ─▶ ranked ─▶ surfaced ─▶ acknowledged
                │ no evidence                                          │              │ action linked
                ▼                                                      │              ▼
             DROPPED (never shown — no risk without evidence)          │          mitigating
                                                                       │              │ condition clears
                                              user dismisses ◀─────────┤              ▼
                                                   │                   │           resolved
                                                   ▼                   │ evidence ages
                                               dismissed               ▼
                                              (tombstone)            stale → re-evaluate
```

| Transition                           | Trigger                        | Owning agent                         | Guard                                         |
| ------------------------------------ | ------------------------------ | ------------------------------------ | --------------------------------------------- |
| → `detected`                         | engine pass over user data     | Domain agent · Decision Intelligence | finds a candidate condition                   |
| `detected` → `evidenced`             | attach evidence                | RecommendationOS                     | **≥1 evidence with source_table (else DROP)** |
| `evidenced` → `ranked`               | scoring                        | RecommendationOS                     | deterministic severity/impact                 |
| `ranked` → `surfaced`                | render                         | Life Model · Advisor context         | not a generic label; passes display gate      |
| `surfaced` → `acknowledged`          | user views                     | surface                              | —                                             |
| `acknowledged` → `mitigating`        | user acts (goal/doc/account)   | RelationshipManager / domain         | a real linked action                          |
| `mitigating`/`surfaced` → `resolved` | evidence no longer supports it | RecommendationOS re-eval             | condition cleared in data                     |
| `surfaced` → `dismissed`             | user dismisses                 | domain writer                        | record tombstone                              |
| any → `stale`                        | evidence freshness elapsed     | freshness rule                       | time-based                                    |

**Invariant:** the `detected → evidenced` guard is the anti-fabrication chokepoint — **no evidence ⇒ no
risk, ever.** The LLM cannot create or advance a risk; it may only _explain_ a surfaced one.

---

## 4. Severity & ranking (deterministic)

Risks are ranked deterministically (not by the LLM): a function of severity (how bad), likelihood (how
probable given the user's data), and impacted domains. Ranking drives ordering on the dashboard and which
risks the advisor may reference as context. The score and its inputs are recorded so ordering is explainable.

---

## 5. Risk ↔ other entities

- **Risk → Recommendation (ACTION):** a surfaced risk should link to the action(s) that would mitigate it.
- **Risk → Goal:** mitigating a risk may propose a candidate goal (user-confirmed, never auto-created).
- **Risk → Relationship:** a risk that spans domains (e.g. a liquidity risk affecting a home goal) must cite
  a real graph edge to claim the cross-domain link (citation contract).
- **Risk → Decision:** decision simulations surface risks of each option with their calculation trace.

---

## 6. Observability

- `detected/dropped` is logged (how many candidates, how many dropped for no evidence) — proves the engine
  isn't fabricating.
- `surfaced` risks carry their evidence + rank + provenance, renderable with a ProvenanceBadge.
- Dismissals + resolutions are tracked so we never re-surface a dismissed risk and can show progress.

---

## 7. Invariants (risk-specific)

1. No risk without evidence (RecommendationOS guard).
2. No archetype/template/generic-label risk (display gates).
3. The LLM never creates or ranks risks; it may only explain a surfaced one (no advice on _how_ to fix —
   that's a recommendation, with its own boundary).
4. A dismissed risk is not re-surfaced unless new evidence materially changes it.
5. Risk severity/ordering is deterministic and explainable.
6. Cross-domain risk claims require a cited real edge.

---

## 8. Failure / escalation

| Failure                                 | Handling                                                    |
| --------------------------------------- | ----------------------------------------------------------- |
| Candidate risk with no evidence         | dropped (not shown)                                         |
| Generic/template label slips through    | display gate hides it; logged as a quality signal           |
| Evidence ages                           | mark stale + re-evaluate, don't keep asserting              |
| Engine can't assess (insufficient data) | honest empty state + Missing-Data nudge, not a guessed risk |

**Open questions:**

1. Is severity scored per domain or globally normalized for the dashboard?
2. When does a resolved risk become a celebrated "progress" signal vs. silently disappear?
3. How are user-dismissed-but-still-true risks handled (suppress vs. periodic gentle re-raise)?

---

## 10. Live vs planned

- **Live:** evidence-or-nothing; no archetype risks; cross-domain citation; grounded Life Model risks.
- **Planned:** documented severity model; mitigation linkage; formal dismissal/resolution lifecycle; per-risk
  freshness + re-evaluation cadence; a "progress" surface for resolved risks.

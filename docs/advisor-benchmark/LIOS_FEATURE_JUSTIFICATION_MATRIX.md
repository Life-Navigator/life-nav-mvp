# LIOS Feature Justification Matrix

**Date:** 2026-06-16
**Purpose:** Force every proposed LIOS component to _earn its complexity_ against the measured advisor benchmark. For each component we ask: what benchmark weakness would it solve, what does the data actually say, and is the case JUSTIFIED, PARTIALLY JUSTIFIED, or UNPROVEN.
**Inputs:** `WIN_LOSS_REPORT.md`, `CLAUDE_CONTROL_EXPERIMENT.md`, `ADVISOR_V3..V6_RESULTS.md`, the 8 criterion analyses (ACTIONABILITY, TRADEOFF_DISCOVERY, DECISION_FRAMING, PERSONALIZATION, QUESTION_QUALITY, UNDERSTANDING, TRUST, EXECUTIVE_PRESENCE), the LIOS corpus (`docs/lios-*`), and the live code (`apps/lifenavigator-core-api/app/`).

---

## The two proven levers (the only things the benchmark says move the score)

The entire residual gap (LN+Gemini V6 **6.66** vs Claude **8.16**, a 7.5 miss) decomposes into exactly two measured causes:

- **(a) Model capability — PROVEN, +0.64 / +1.5.** Swap _only_ the model behind a byte-identical pipeline (Gemini-2.5-flash → Claude Opus, `USE_VERTEX_CLAUDE`) and the score jumps **6.66 → 7.30** all-50, **8.08 on the 44 non-fallback turns** (= parity with raw Claude 8.00). `CLAUDE_CONTROL_EXPERIMENT.md` Q1/Q3. Claude improved _every_ quality criterion (actionability +1.5, insight +0.9, framing +0.6, etc.).
- **(b) The number-gate fallbacks — PROVEN, ~0.7 pt, recoverable.** "Almost the entire 0.70-point platform cost is the VALIDATOR's number gate, not the other layers." On the 6 fallback turns LN+Claude loses ~1.7 each; clear them and LN+Claude ≈ 8.0. `CLAUDE_CONTROL_EXPERIMENT.md` Q2/Q4. The V5→V6 arc confirms it from the Gemini side: graceful degradation (repair-not-reject) cut fallbacks 12→5 and lifted 5.83→6.66 with zero trust cost.

**The governing rule for this matrix:** _a component that does not address (a) model capability or (b) the fallback/number-gate mechanism cannot, on the current evidence, move the benchmark — so it is at best UNPROVEN against it._ `CLAUDE_CONTROL_EXPERIMENT.md` Q5: "Neither an orchestrator, agents, critics, nor multi-agent execution touches either lever. LIOS is not on the critical path to advisor quality." The memory verdict is the same: **build LIOS-Lite, not LIOS as designed.**

**Prod-existence note (from live code + runtime-blueprint CURRENT_STATE_AUDIT):** Relationship Manager (`services/relationship_manager.py`), Recommendation Engine (`routers/recommendations.py` + RecommendationOS, migration 148), Compliance/validator (`services/advisor_validator.py` + `agents/trust_safety.py`), Memory/context (`grounding/context_builder.py` + `analytics.advisor_turns`), GraphRAG (3-store + `services/graphrag_*`), and the Scenario engines (`services/scenario_compare.py`, `scenario_tree.py`) **already exist in prod**. The live `agents/orchestrator.py` is a _single-agent, single-LLM-call_ grounded flow — it is **not** the LIOS multi-agent orchestrator. **Critic and per-domain Domain Agents do not exist** (no `critic.py`; domains are deterministic). So "would LIOS help?" mostly reduces to "would _adding the speculative multi-agent layer_ help?" — and the answer the data gives is: not against this benchmark.

## The honest bottom line

1. **The two proven levers are not LIOS.** (a) Model capability — selectively route high-stakes turns to Claude (`USE_VERTEX_CLAUDE`), Gemini for the rest (cost/latency dictate _where_, not _whether_). (b) The fallback/number-gate — graceful degradation already banked +0.83 (V6); finish the k-notation/false-positive fixes. Doing only these gets LN+Claude to ~8.0 on passed turns with **0 fabrications**.
2. **Most LIOS components already exist** (RM, Recommendation, Compliance, Memory, GraphRAG, Scenario) and were _held constant_ in the Claude control experiment, where the pipeline retained **91% of raw-Claude quality (100% on passed turns)** — proving they add **no quality drag and no measured quality lift**. They are plumbing, not levers.
3. **The genuinely new, speculative LIOS pieces** (multi-agent Orchestrator routing, the 4-call Decision pipeline, per-domain Domain Agents, separate Tradeoff/Conflict/Recommendation LLM agents, the Critic) are **UNPROVEN or recommended-against**: none touches lever (a) or (b), each adds latency (serial decision tail → 68s p95) and cost ($4/day breached at ~46-100 users vs ~400), and the simulation's own verdict is **"do NOT build LIOS as designed."**
4. **The one defensible build-LIOS bet** is _measurement-gated_: before building the Decision Engine, run the cheap coverage experiment (multi-agent vs single advisor on **data-rich** personas, multi-turn, loaded context). That is also the only test that can finally answer the GraphRAG/Memory/personalization question the cold benchmark left open. Build the expensive pieces **only if that measurement shows a real gain** — otherwise LIOS-Lite (deterministic spine + observe-only wrap + intent routing + prompt composition + Discovery Analyst + ONE high-stakes Decision/Critic call) is the 80/20 answer.

**Make LIOS earn its complexity: on the current evidence, only the model-routing and fallback-fix levers are paid for. Everything else is on credit until a coverage measurement clears it.**

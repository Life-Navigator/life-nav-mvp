# Pilot Journey Audit — Monday 20-person Pilot

**Scope:** complete user flow, web (lifenavigator.tech) + core-api v125. **Audit only — no code changed.**
**Date:** 2026-06-18. **Method:** every finding grounded in real code (`file:line`). Brutal where warranted; this feeds Go/No-Go.

## How the flow actually wires (verified)

The real, gated sequence is enforced by `apps/web/src/proxy.ts` (the Next middleware — note: there is **no** `middleware.ts` at app root; `dashboard/page.tsx:14` claims "gating happen in middleware", which is technically `proxy.ts`):

1. Magic link → `/auth/confirm` (`MagicLinkPanel.tsx:37` sets `next=/onboarding`).
2. `auth/confirm/route.ts:93` — if `!setup_completed`, **redirects to `/onboarding/financial-profile`** (NOT `/onboarding`, NOT the advisor). The `next=/onboarding` hint is overridden here.
3. `/onboarding/financial-profile` = `SampleFinancialProfile` (persona picker). On activate → `router.push('/dashboard/advisor?onboarding=1')` (`SampleFinancialProfile.tsx:75`).
4. Advisor discovery (`dashboard/advisor/page.tsx`). On finish → `DiscoveryReveal` → `/dashboard`.
5. `proxy.ts:168-176`: `!setup_completed`→financial-profile; `setup_completed && !onboarding_completed`→advisor.

So the advisor flow **is** the onboarding, but the user's first screen after login is a **sandbox persona picker**, not a welcome or the advisor. That framing is the single biggest first-impression risk (see trust-breaks).

## Cross-cutting / navigation

- **`/settings/integrations` is referenced 30+ times across OAuth start + all 3 callbacks but the route does not exist.** This is the leftover the brief warned about. Any OAuth touch that isn't a clean success → 404. (`oauth/**`, see Step 9.)
- **Two email surfaces disagree:** `/dashboard/email` = "Coming soon" (pilot-correct); `/email` = full inbox app (off-message, reachable by URL). `Sidebar.tsx` email dropdown (lines 882-994) is dead code (no `hasDropdown` nav item) but still references `/email` and `/dashboard/integrations`.
- **`/dashboard/integrations`** renders `IntegrationsOverview` (Connect buttons) and is reachable by direct URL though removed from primary nav (`Sidebar.tsx:425-466` comment). Its connects route through the same broken `/settings/integrations` default. ~40 dashboard routes exist; primary nav exposes 12 — many (`scenario-lab`, `decision-brain`, `compare-futures`, `next-dollar-optimizer`, `military`, `benefits`, etc.) are reachable only by URL and were not in scope but are unvetted for the pilot.
- **Onboarding doc drift:** `proxy.ts:130-140` lists 6 "legacy" onboarding surfaces (questionnaire/interactive/hub/review/sections/converse) folded into the canonical flow; the components still exist (`components/onboarding/*Questionnaire.tsx`). Confirm none are reachable.

## TOP trust-breaks — MUST FIX before Monday

1. **Calendar "Connect" dead-ends on a 404.** OAuth creds unprovisioned → not-configured branch → redirect to nonexistent `/settings/integrations`. Even after creds land, every OAuth _error_ path 404s. Either (a) hide/disable Calendar Connect for the pilot (mirror the Email "Coming soon" treatment), or (b) provision creds **and** repoint all OAuth fallbacks from `/settings/integrations` → `/dashboard/integrations` (or `/dashboard/calendar`). `oauth/callback/{google,microsoft,linkedin}/route.ts`, `oauth/{google,microsoft}/route.ts`, `calendar/page.tsx:37`.
2. **The full email app at `/email` contradicts "Email coming soon."** A pilot user who finds it sees Connect/inbox UI that can't work. Gate `/email` behind the same coming-soon state or block the route. `app/email/page.tsx`.
3. **First screen after login is a fictional "sample financial profile" picker.** This reads as a demo and undercuts trust before the advisor ever speaks. Add a one-line framing ("Beta uses safe sample data so you can explore the full system") _above the fold on the first screen_, and/or a welcome interstitial explaining why. `SampleFinancialProfile.tsx:146-152`.
4. **Silent feedback loss.** `/api/feedback/pilot` returns `{ok:true}` even when core-api drops the write. The pilot's core deliverable is the feedback signal — you can't trust the rollup. Surface a real failure (or at minimum log + alert). `api/feedback/pilot/route.ts:25-28`.
5. **Advisor opening turn can fail with an empty chat (intermittent 502).** First impression of the product's headline feature. Confirm the blocking fallback path is solid and the Retry on the _opening_ turn re-fires `send('', null)`. `advisor/page.tsx:383-385,351-361`.

## SHOULD FIX (high-value, not strictly blocking)

6. **Reveal 14s auto-advance** races the user reading the narrative and the feedback prompts — pause/extend when feedback is visible, or make advance explicit-only. `DiscoveryReveal.tsx:132-141`.
7. **Life Graph full-screen view has no in-view "back to dashboard."** Add an exit affordance. `explainable/page.tsx:110-137`.
8. **Reports hub offers 6 reports that will be near-empty for sandbox users.** Hide reports with no data, or label "needs more data." `reports/page.tsx:8-39`.
9. **Recommendations empty-state copy says "upload documents,"** but pilot onboarding is persona-based — align the copy. `recommendations/page.tsx:484-487`.
10. **Inline reveal chips state inferences as fact with no confirm/correct.** Add a quick "not quite" on the chip. `advisor/page.tsx:515-549`.

## What's genuinely strong (don't touch)

The trust spine (no fabrication, honest empties, source labels) holds across the app — the breaks are at the **edges** (OAuth fallbacks, the parallel `/email` app, the sandbox-first first impression, silent feedback drop), not the core intelligence surfaces.
